To guide this investigation, we pose the following research questions:RQ1: How do developers integrate LLMs into their real-world software development workflows, and what forms of contribution do these models make?  To address this question, we manually annotated 500 unique code comments that explicitly referenced LLM usage and extended the analysis to 12,043 comments using LLM-based annotation consolidated through the Dawid-Skene (DS) Expectation-Maximization (EM) framework [8]. We categorized each instance along two dimensions: Task Type, describing the software engineering activity supported by the LLM (e.g., code implementation, enhancement, testing, documentation, or debugging); and LLM Contribution Type, representing how the model assisted developers (e.g., implementation, knowledge and concept support, or artifact generation). Using both quantitative and qualitative analyses, we find that LLMs are most often used for code implementation and conceptual guidance, positioning them as active collaborators within real-world software development workflows.RQ2: How do developers’ commit-level activities reflect the software engineering actions involved in integrating and modifying LLM-assisted code?  Each annotated comment was linked to its corresponding introductory commit. We also linked the first change commit, which refers to the earliest subsequent commit that modifies, extends, or deletes the LLM-assisted code introduced in the introductory commit. Using semantic clustering of commit messages with HDBSCAN [32], we identified 11 integration and 9 modification activity types. We found that developers most often use LLMs for feature development, project initialization and setup, and code refactoring. During modification, the dominant activities were code refactoring and feature development, followed by bug fixing and repository management.RQ3: How has developers’ LLM usage behavior evolved over time?  We performed a longitudinal analysis of GitHub commits from December 2022 to July 2025 to analyze the evolution of LLM-assisted activities at two levels of granularity from RQ1: a) Task Types and b) LLM Contribution Types. Over time, code implementation remained the dominant activity but gradually declined in relative frequency, while enhancement and documentation gained prominence. At the contribution level, implementation continued to lead,while knowledge-seeking behavior increased most rapidly, suggesting that developers are progressively engaging LLMs for conceptual reasoning, design exploration, and informed decision-making rather than solely for code generation.Contributions. We used Hierarchical Density-Based Spatial Clustering of Applications with Noise (HDBSCAN) [32] to group semantically similar commit messages. HDBSCAN can identify clusters of different densities and label uncertain points as noise, without requiring a predefined number of clusters. This method has been adopted in software engineering research, including for analyzing developer comments [26].  Each commit message underwent text normalization, lemmatization, and vector embedding using the all-MiniLM-L6-v2 model [42]. After manual parameter sweeping, we selected cosine distance (?? = 0.20), minimum cluster size (???????????? = 10), and the leaf cluster selection. This procedure produced 58 distinct clusters.  We interpreted and labeled each cluster through a structured process that combined keyword heuristics with human review. We first built a keyword dictionary based on existing commit-message taxonomies [22], including categories such as feature, fix, refactor, documentation, and build. We then expanded the dictionary by extracting additional keywords from our dataset using TF-IDF weighting and part-of-speech filtering to capture frequent action-oriented terms such as security, revert, dependency, style, and optimization.  For each cluster, we matched the most significant tokens against the keyword dictionary using a weighted scoring function that emphasized key verbs and nouns such as add, fix, refactor, and upgrade. We then assigned preliminary labels and refined them through manual review. One author examined representative messages per cluster to ensure conceptual accuracy and adjust the labels as needed. Finally, we merged related clusters into broader categories [12]. For example, Refactoring, Code Cleanup, and Refactor After Testing were combined into a single cluster labeled Code Refactoring.  After completing the initial labeling and merging phase, a second annotator independently reviewed the consolidated clusters to verify label consistency and address remaining ambiguities. Our clustering identified 11 categories of developer intent involving LLM assistance. These categories are: Feature Development, Project Initialization and Setup, Code Refactoring, Documentation, Repository Management, Security and Compliance, Release and Deployment, Tooling, Bug Fixing, LLM/AI Work, and Testing and Verification.2.4.2 Identifying Modification Activities on LLM Assited Code. However, a significant and growing portion of their interaction with LLMs involves Knowledge and Concept Support. Combined with RQ3, which shows that knowledge- and reasoningoriented usage has been steadily increasing over time, this indicates that developers increasingly treat LLMs not merely as code generators but as conceptual collaborators. Future coding assistants should therefore expand beyond producing code to providing adaptive reasoning support-detecting developer intent, surfacing explanations, and addressing knowledge gaps during development. For instance, comment patterns such as “ChatGPT suggested this fix and it seems to work” indicates uncertainty uncertainty about solution validity, suggesting assistants should also provide brief reasoning or impact explanations within the project context.  Results from RQ2.A show that LLMs are primarily integrated during feature development and project setup, where they accelerate early development. However, RQ2.B reveals that this LLM-aided code often requires subsequent human modification, mainly through refactoring, debugging, and standardization. This highlights the need for assistants that can adapt to project context by leveraging repository history, dependency structure, and coding conventions to produce code that better aligns with existing architectures and reduces manual rework. They could assess generated code against project tests and conventions to prevent integration errors.Overall, future of LLM-assisted development should emphasize:(1) expanding from code generation to knowledge and reasoningassistance;(2) integrating project- and repository-specific learning for contextaware generation; and(3) enabling feedback-driven refinement where LLMs learn from developer edits.  By addressing these directions, coding assistants can evolve from isolated generators to adaptive collaborators that enhance both productivity and understanding throughout the software lifecycle.5	Related WorkWe divide the related work into two parts: (1) code comments and commit messages to infer developer intent and activities, and (2) how LLMs are used in software development.Code Comments and Commit Messages. Construct validity concerns the extent to which the methods and measures used in the study accurately capture the intended research constructs.Our detection of LLMrelated comments relied on keyword-based queries, which may introduce false positives (e.g., unrelated names like “Claude”). We mitigated this by manually validating a random subset for precision. Annotation bias was reduced through independent labeling by multiple annotators, iterative guideline refinement, and inter-annotator agreement checks.Internal validity. Internal validity refers to whether the observed relationships genuinely reflect causal or meaningful associations, rather than being influenced by confounding factors or methodological artifacts. The linkage between LLM-referenced comments and commits assumes repository histories preserve temporal order and authorship. We excluded commits with anomalous timestamps or inconsistent metadata. Because commits occur at the file level andmay include unrelated changes, we limited analysis to the commit where the LLM-referenced comment first appeared and the earliest subsequent commit modifying the same file.External validity. External validity concerns how well our findings generalize beyond the specific dataset, repositories, or environments analyzed. Our dataset focuses on public GitHub repositories in Python and JavaScript, which may not represent other ecosystems or closed-source projects. Moreover, it captures only self-admitted LLM usage, explicit mentions of tools like ChatGPT or Copilot, thus underrepresenting silent or unacknowledged use. As LLM-assisted development practices evolve rapidly, our findings should be viewed as representative of current open-source trends rather than the full spectrum of developer behavior.7	Conclusion and Future WorkThis study investigates the range of development tasks where developers rely on LLMs, the forms of contribution these models provide, and how LLM-assisted code is integrated and adapted within real-world software projects. By linking LLM-referenced code comments to their corresponding commits, we provide empirical evidence of how human–AI collaboration unfolds in practice.  Developers use LLMs across diverse software engineering activities, most frequently for implementing new features, generating code snippets, and automating routine tasks such as testing and documentation. Beyond direct code generation, LLMs also contribute conceptually—offering guidance, explanations, and alternative design ideas that help developers reason about implementation choices. Once integrated, however, much of this AI-assisted code undergoes human refinement through refactoring, debugging, and reorganization to meet project-specific standards and maintainability requirements. Together, these patterns reveal an evolving collaboration where LLMs accelerate creation and ideation, while developers continue to ensure reliability and alignment within real-world projects.  Future work should focus on enhancing LLMs’ ability to provide adaptive, context-aware assistance that identifies when developers require conceptual support. Equally important directions include assessing the long-term quality and evolution of LLM-generated code, investigating how models can learn from developer modifications, and extending such analyses to larger open-source and industrial ecosystems. Advancing these research fronts can further evolve LLMs into proactive and context-sensitive partners that continuously improve collaboration between automation and human expertise in software engineering.References[1] Eman Abdullah AlOmar, Anushkrishna Venkatakrishnan, Mohamed Wiem Mkaouer, Christian Newman, and Ali Ouni. 2024.
